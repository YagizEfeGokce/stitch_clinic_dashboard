Section 1

1. Introduction

1.1 Purpose of this Document

This Software Architecture Document (SAD) describes the architecture of Dermdesk, a multi-tenant Clinic Relationship Management (CRM) SaaS platform for aesthetic clinics. The document is produced as part of the Software Architecture bachelor's course project and follows the 4+1 architectural view model. Version 1 covers the Use Case View, Logical View, and Process View as required for Phase 1.

1.2 Scope

Dermdesk is a web-based application that allows aesthetic clinics to manage appointments, clients, staff, inventory, finances, and marketing through a single dashboard. The system also exposes a public booking page per clinic, enabling clients to schedule appointments without creating an account.

1.3 Architecture Style

The system follows a Layered Single-Page Application (SPA) architecture combined with a Backend-as-a-Service (BaaS) model:

  • The frontend is a React SPA structured into distinct layers: Presentation, Business Logic, and Data Access.
  • The backend is provided by Supabase (PostgreSQL database, Auth service, Row-Level Security policies, Realtime subscriptions, and Storage).
  • Communication between the frontend and backend occurs via RESTful HTTP (PostgREST) and WebSocket (Realtime channels).
  • Multi-tenancy is enforced at the database level using Row-Level Security (RLS) policies scoped to each clinic's unique ID.

2. System Selection

2.1 Target System Overview

The selected system is Dermdesk — a full-stack CRM and operational management platform designed specifically for aesthetic clinics in Turkey. The platform is delivered as a web SaaS product accessible from any modern browser.

2.2 System Purpose

Dermdesk removes the operational burden from clinic owners by centralising appointment scheduling, client records, staff management, inventory tracking, and financial reporting in a single interface. The system also provides each clinic with a shareable public booking link so clients can self-book appointments at any time.

2.3 Users and Stakeholders

2.4 Core Functionalities

  • Appointment Scheduling – visual daily/weekly calendar with drag-and-drop rescheduling
  • Client Management – full client profiles including history, photos, medical notes, and physical metrics
  • Public Booking – per-clinic booking page accessible without authentication via /book/:clinicSlug
  • Service Catalogue – configurable treatments with pricing, duration, and material requirements
  • Inventory Tracking – stock management with automatic deduction on appointment completion
  • Financial Reporting – revenue charts, transaction ledger, KPI cards
  • Team Management – invite staff by email, assign roles, set availability schedules
  • Performance Analytics – staff KPIs, client growth charts, campaign effectiveness
  • Role-Based Access Control (RBAC) – route-level and data-level enforcement
  • Multi-Tenancy – complete data isolation between clinics via PostgreSQL RLS

3.4 Partial UI Code – Booking Wizard

The BookingWizard component (src/components/booking/BookingWizard.jsx) implements UC2 (Manage Appointments). It is a five-step multi-screen wizard: Client Selection → Service Selection → Date & Time → Staff Selector → Booking Summary. The snippet below shows the state management, step navigation, and appointment submission logic. 
// src/components/booking/BookingWizard.jsx (excerpt)
import { useState } from 'react';
import { useNavigate } from 'react-router-dom';
import { supabase } from '../../lib/supabase';
import { useAuth } from '../../context/AuthContext';
import { useToast } from '../../context/ToastContext';

const TOTAL_STEPS = 5; // Client → Service → DateTime → Staff → Summary

export default function BookingWizard() {
  const navigate = useNavigate();
  const { clinic } = useAuth();
  const { success, error: toastError } = useToast();

  const [step, setStep] = useState(1);
  const [isSubmitting, setIsSubmitting] = useState(false);
  const [data, setData] = useState({
    client: null, service: null,
    date: null, time: null, staffId: null
  });

  // Advance wizard and merge new step data into shared state
  const handleNext = (newData) => {
    setData(prev => ({ ...prev, ...newData }));
    setStep(prev => prev + 1);
  };

  const handleConfirm = async () => {
    if (!data.client || !data.service || !data.time) return;
    setIsSubmitting(true);

    try {
      // Auto-assign first available staff when none is explicitly chosen
      let finalStaffId = data.staffId;
      if (!finalStaffId && clinic?.id) {
        const { data: available } = await supabase.rpc('get_available_staff', {
          p_clinic_id: clinic.id,  p_date: data.date,
          p_time: convertTo24Hour(data.time),
          p_duration_min: data.service.duration_min || 30
        });
        const first = available?.find(s => s.is_available);
        if (first) finalStaffId = first.staff_id;
      }

      const { error } = await supabase.from('appointments').insert([{
        client_id:    data.client.id,
        service_id:   data.service.id,
        service_name: data.service.name,
        staff_id:     finalStaffId,
        date:         data.date,
        time:         convertTo24Hour(data.time),
        status:       'Scheduled',
        clinic_id:    clinic.id,
      }]);

      if (error) throw error;
      success('Randevu başarıyla oluşturuldu!'); // "Appointment created successfully!"
      navigate('/schedule');
    } catch (err) {
      toastError('Randevu oluşturulamadı. Lütfen tekrar deneyin.');
    } finally {
      setIsSubmitting(false);
    }
  };
}
Architecture note: The wizard follows a unidirectional data flow pattern — each child step component calls handleNext(data) to pass its result upward. The parent accumulates all step data before the final Supabase insert call on confirmation. This keeps step components stateless and independently testable.

4.5 Partial Backend Code – Appointments API

The AppointmentsAPI class (src/lib/api/appointments.js) wraps all Supabase queries for the appointments table. The snippet below shows the conflict-aware createAppointment and the stock-validation logic triggered when completing an appointment. 
// src/lib/api/appointments.js (excerpt)
class AppointmentsAPI extends BaseAPI {

  /** Creates an appointment with optional double-booking guard */
  async createAppointment(appointmentData, checkConflicts = false) {
    try {
      if (checkConflicts) {
        const { data: conflicts } = await supabase
          .from('appointments')
          .select('id')
          .eq('clinic_id', appointmentData.clinic_id)
          .eq('date', appointmentData.date)
          .eq('time', appointmentData.time)
          .not('status', 'eq', 'Cancelled');

        if (conflicts?.length > 0)
          return { data: null, error: 'Bu tarih ve saatte başka bir randevu mevcut' };
      }
      return this.create(appointmentData);
    } catch (error) {
      return { data: null, error: handleError(error) };
    }
  }

  /** Validates inventory stock before marking an appointment Completed */
  async checkStockForCompletion(appointmentId) {
    const { data, error } = await supabase
      .from('appointment_materials')
      .select(`quantity_used, inventory:inventory_item_id(stock, name, unit)`)
      .eq('appointment_id', appointmentId)
      .eq('deducted', false);

    if (error) return { error: 'Stok kontrolü yapılamadı', items: [] };

    const insufficient = data
      .filter(m => m.inventory?.stock < m.quantity_used)
      .map(m => ({ name: m.inventory.name, current: m.inventory.stock,
                   required: m.quantity_used, unit: m.inventory.unit }));

    return insufficient.length > 0
      ? { error: `Yetersiz stok: ` + insufficient.map(i =>
            `${i.name} (${i.current} mevcut, ${i.required} gerekli)`).join(', '),
          items: insufficient }
      : { error: null, items: [] };
  }
}

5.4 Code Snippets – Authentication Context

The AuthContext (src/context/AuthContext.jsx) subscribes to Supabase's auth event stream and exposes the authenticated user, role, profile, and clinic to the entire component tree. The snippet below shows the event listener and the refreshUserData function that loads profile data after sign-in. 
// src/context/AuthContext.jsx (excerpt)
const { data: { subscription } } = supabase.auth.onAuthStateChange(
  async (event, session) => {
    if (!mounted) return;

    if (['SIGNED_IN', 'INITIAL_SESSION'].includes(event)) {
      setUser(session?.user ?? null);
      if (session?.user)
        refreshUserData(session.user.id); // loads profile + clinic
    } else if (event === 'SIGNED_OUT') {
      setUser(null); setRole(null);
      setProfile(null); setClinic(null);
    }
    setLoading(false);
  }
);

const refreshUserData = async (userId) => {
  const { data: profile } = await supabase
    .from('profiles').select('*')
    .eq('id', userId).maybeSingle();

  if (profile) {
    setProfile(profile);
    setRole(profile.role);          // 'owner' | 'admin' | 'doctor' | 'staff'

    const { data: clinic } = await supabase
      .from('clinics').select('*')
      .eq('id', profile.clinic_id).maybeSingle();

    if (clinic) setClinic(clinic);
  }
};
